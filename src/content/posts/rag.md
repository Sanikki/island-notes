> **RAG 不是让大模型“变聪明”，而是让大模型“会翻书”。**
> 就像考试的时候，虽然你已经努力的准备了，但还是要准备一份小抄，以备不时之需。
> **核心矛盾：** 大模型什么都“知道”，但什么都“记不准”。RAG 用外部知识库，把“凭记忆瞎猜”变成“开卷考试”。

# RAG 是什么？

**RAG（检索增强生成）** = **检索（Retrieval）** + **增强（Augmented）** + **生成（Generation）**。

| **阶段**         | **江湖比喻**               | **做什么**                                     |
| ---------------------- | -------------------------------- | ---------------------------------------------------- |
| **索引（入库）** | **藏经阁整理藏书**         | **把文档拆块、转成向量、存进向量库**           |
| **检索（召回）** | **大侠去藏经阁翻书**       | **把用户问题转成向量，去向量库里找最像的几段** |
| **增强（注入）** | **把找到的书页夹在奏章里** | **把检索到的原文拼进 Prompt 上下文**           |
| **生成（回答）** | **大侠看完书后回话**       | **大模型基于“问题 + 书页内容”生成答案**      |

**核心原则：** 检索决定了 RAG 的“上限”，生成决定了 RAG 的“下限”。**菜买对了，怎么做都好吃；菜买错了，食神来了也白搭。**

**RAG概念的误区**

| **误解**                 | **真相**                                                 |
| ------------------------------ | -------------------------------------------------------------- |
| **“RAG 就是向量检索”** | **向量检索只是“找”，RAG 是“找 + 读 + 答”三件事。**   |
| **“RAG 能取代微调”**   | **微调改“记忆”，RAG 给“参考书”，各管各的。**         |
| **“RAG 跑通就行”**     | **RAG 是系统工程，每个环节都能玩出花来，也能玩出坑来。** |

# RAG 核心参数（三把尺子）

### Chunk Size（块大小）

* **是什么**：把长文档切成的每一块的“厚度”（通常按 Token 计，如 512、1024）。
* **太大**：一块里塞了太多信息，关键内容被淹没，像大海捞针。
* **太小**：语义被切断，只有“左脚尖”没有“轻功心法”，上下文不完整。

**口诀：短句问答用 256，技术文档上 1024；小说剧本用 2048，先跑 Eval 再说话。**

### Overlap（重叠）

* **是什么**：前一块的尾部内容复制若干行到下一块的开头，形成“装订线”。
* **为什么**：防止关键口诀恰好被切在两块之间，导致两块都搜不到完整语义。
* **江湖经验**：通常设为 **Chunk Size 的 10%~20%**（如 Size=512，Overlap=50~100）。

### Top-K

* **是什么**：一次从藏经阁搬几本书出来看。
* **太小（K=1）**：只看一本，万一这本写错了，大侠就被带偏了。
* **太大（K=10）**：搬了一堆书，里面有大量废话（噪音），大模型上下文窗口被塞爆，抓不住重点。
* **江湖经验**：通用场景 **K=3~5**；复杂推理放宽到 10 但要配合 Rerank。

# RAG 进阶五件套

### 查询改写（Query Rewriting）

* **痛点**：用户问“那个不咋地的东西咋退啊？”——口语化、模糊，向量搜不准。
* **解法**：先让大模型当“翻译官”，把口语转成正规检索语：“如何申请商品退货退款流程？”
* **价值**：架起“人话”和“数据库语言”之间的桥梁。

### 元数据过滤（Metadata Filtering）

* **痛点**：向量库里有去年财报、今年财报、公开文档、机密文档，混在一起搜。
* **解法**：给每个文档块打标签（`author`、`date`、`department`、`access_level`），检索时先按规则砍掉不相关的（`WHERE date >= '2025-01-01' AND department='研发部'`），缩小范围后再做向量检索。
* **价值**：**防泄密 + 防穿越 + 提效。** 这是 RAG 系统最重要的“守门员”。

### 混合检索（Hybrid Search）

* **痛点**：纯向量检索懂“意思”但不懂“精确匹配”（如产品编号 `SN:89757-001`）。纯关键词检索懂“精确匹配”但不懂“同义词”。
* **解法**：**两条腿走路**。同时跑向量检索（语义）和关键词检索（BM25/TF-IDF），合并结果、去重、综合打分。
* **价值**：既能识别“智能手机”和“iPhone”的语义关系，又能精准命中“序列号”。

### 重排序（Rerank）

* **痛点**：Hybrid Search 搜回来 20 条，里面有 10 条沾边但排序很乱。直接用 Top-3，可能真正有用的排第 5。
* **解法**：先粗召回 20 条，再用一个更精准的 **Rerank 模型**（如 Cohere Rerank、BGE Reranker）逐一做交叉注意力计算，重新打分排序，只取 Top-3。
* **价值**：**用粗排的计算量换精排的准确率，极大降低幻觉。**

### 多轮检索（Multi-turn Retrieval）

* **痛点**：复杂问题（如“张三丰和郭靖谁的武功高？”）第一轮搜到的都是两人生平，缺对比信息。
* **解法**：大模型看完第一轮结果，发现缺信息，**自动提炼新关键词**发起第二轮检索。多轮下来，补全所有缺失信息。
* **价值**：**解决需要多跳推理（Multi-hop）的复杂问题。**

# 诊断篇（出问题如何排查）

1. **黄金法则（永远排第一）**

**把召回的 Top-K 原始 Chunk 打印出来。**

**不看原始材料就调 Prompt = 闭着眼睛修水管。**

1. **两步断案法**

**第一步：验召回**

* **打印出召回的 Chunk**
* **问自己：****“我是人，看了这些材料，能正确回答问题吗？”**
* **不能 → ****召回问题**
* **能 → 可能是生成问题，走第二步**

**第二步：验生成**

* **把正确答案所在的 Chunk（人工翻到的）“硬编码”塞进上下文**
* **模型答对了 → ****召回问题**（铁证如山，材料在却捞不出来）
* **模型答错了 → ****生成问题**（材料给了，但没用好）

1. **快速定性表**

| **现象**                                    | **诊断**             | **动作**                                                    |
| ------------------------------------------------- | -------------------------- | ----------------------------------------------------------------- |
| **打印出来的材料里没有答案**                | **召回问题**         | **调 Chunk Size / 换 Embedding / 加元数据 / Hybrid Search** |
| **材料里有答案，模型胡言乱语**              | **生成问题（指令）** | **精简 System Prompt + 强制原文摘录 + Schema**              |
| **材料里有答案但被无关材料淹没**            | **召回问题（精度）** | **调大 Top-K + 加 Rerank（把正确答案挤到前面）**            |
| **材料里有多个观点，模型缝合出怪话**        | **生成问题（推理）** | **降 Temperature + 明确“矛盾时指出分歧”**                 |
| **打印出来的材料里有正确答案，但排第 7 名** | **召回问题（排序）** | **加 Rerank，让正确答案进入 Top-3**                         |

# 召回问题 vs 生成问题

**先把“菜买对”，再去研究“火候”和“调料”。这是我在生产环境滚了无数次钉板后，唯一的“免死金牌”信条。--DeepSeek**

### 1. 因果逻辑的“绝对优先级”（垃圾进，垃圾出）

**生成环节的“答案”完全依赖于召回环节喂进去的“上下文（Context）”。**

* **修召回，就是在修上下文，只要上下文对了，即使LLM效果差一点，但依旧能回答出来。**
* **修生成，就是修逻辑，即使上下文存在问题，LLM也能“恰圆其说”。**

**结论：召回是“0”和“1”的问题（有没有），生成是“1”到“100”的问题（好不好）。先确保“有”，再谈“好”。**

### 2. 调试与排查的“难易程度”（白盒 vs 黑盒）

* **修召回（白盒操作）**：召回错了，你能拿到明确的证据。你可以去向量库里查 Top-k返回的 Chunk，看到底是 Chunk Size 没切好，还是 Hybrid Search 权重没调对，还是 Metadata 过滤条件写死了。**你能截图给老板看：“你看，系统根本没把正确答案捞出来！”** 这是可以量化、可视化排查的。
* **修生成（黑盒操作）**：生成错了，你怎么查？你只能看着大模型胡说八道，去调 `Temperature`，去改“提示词（Prompt）”，甚至考虑去“微调（SFT）”。**改十次 Prompt，可能错法完全不一样，完全不可控。**

**结论：修召回是用“尺子”量，修生成是用“手”去抓水。量尺子总比抓水容易。**

### 3. 理由三：修复成本的“杠杆效应”（投入产出比）

* **修召回（一次投入，全局受益）**：你优化了 Metadata 过滤，或者修好了 Embedding 模型。**这****一个改动，会让整个知识库的检索精度全面提升**，无论用户问什么都能吃到红利。这是“一劳永逸”的基建工程。
* **修生成（特事特办，顾此失彼）**：你为了让 Agent 回答“退货流程”写了一大段精妙的 Prompt 修正。结果用户转头去问“优惠券使用”，这段 Prompt 可能完全失效，甚至引发新的冲突。**修生成往往是对特定场景的“打补丁”，修不完的。**

**结论：花同样的一天时间，修召回能解决 100 个问题，修生成可能只解决了 1 个问题，还制造了 2 个新问题。**

### 4. 极端反例（什么时候才先修生成？）

**当然，武林规矩也有例外。如果出现以下情况，才要考虑先修生成：**

* **召回明明已经把正确答案捞回来了（Top-1 就有标准答案），但大模型愣是读不懂，非要自己瞎编。** 这叫“捧着金饭碗要饭”——此时才说明上下文没问题，**是大模型的阅读理解能力（生成/推理）出了大问题**，需要去修 Prompt 或者换模型。

# 避坑指南

| **坑**                    | **解法**                                                                               |
| ------------------------------- | -------------------------------------------------------------------------------------------- |
| **Chunk Size 拍脑袋瞎设** | **从 512 Token 起步，用 Eval 验证，用数据说话。**                                      |
| **Overlap 设成 0**        | **强制设至少 10% 的重叠**，否则关键词在边界被切断会永久丢失。                          |
| **Top-K 固定不变**        | **简单问答 K=3~5，复杂跨文档推理 K=10+Rerank。**                                             |
| **元数据过滤过严或过松**  | **只保留 8~10 个核心字段；用“模糊过滤 + 精准检索”组合。**                            |
| **只有向量检索**          | **一定要加关键词检索（Hybrid Search）**，否则序列号、产品编号这种精确词会全军覆没。    |
| **召回结果直接喂大模型**  | **一定要加 Rerank**，把最相关的内容挤到前 3 位，否则高质量结果排第 10 就永远出不了头。 |
| **回答出错直接调 Prompt** | **先打印召回材料，人工判卷！**如果材料错了，调 Prompt 是浪费时间。                           |
